# JOB-006 - Dashboard Screen

## Goal
Build the main dashboard screen that summarizes the user’s financial status using mock transaction data.

## Why
The dashboard is the main screen of the personal finance tracker. It should give users a quick overview of their balance, income, expenses, category spending, and recent transactions.

This task turns the placeholder home tab into the first real app experience after login.

## Scope
This task should only focus on:
- Building the dashboard screen UI.
- Showing total balance, total income, and total expenses.
- Showing category-based spending overview.
- Showing a recent transactions preview.
- Using existing mock data, types, utilities, and base UI components.
- Adding subtle Reanimated entrance animations if Reanimated is already available.

This task should not include:
- Transaction list screen implementation.
- Transaction detail screen implementation.
- Add transaction form implementation.
- Real CRUD behavior.
- Backend integration.
- Auth logic changes.
- New package installation.
- Major navigation changes.

## Requirements
- Replace the dashboard placeholder screen with a real dashboard UI.
- Use mock transactions from the existing mock data files.
- Use existing utility helpers for:
  - total income
  - total expenses
  - balance
  - category totals
  - currency formatting
  - date formatting
- Display a main balance card.
- Display income and expense summary cards.
- Display a category spending section.
- Display a recent transactions section.
- Recent transactions section should show only a limited number of latest transactions.
- Each recent transaction item should show:
  - transaction title
  - category name
  - date
  - amount
  - income or expense visual state
- Use existing base components such as:
  - `Screen`
  - `AppText`
  - `Card`
  - `PrimaryButton` only if needed
- Create small reusable dashboard components only when they improve readability.
- Possible dashboard components:
  - `SummaryCard`
  - `BalanceCard`
  - `CategorySpendingCard`
  - `RecentTransactionItem`
- Keep the UI modern, clean, premium, and easy to scan.
- Use consistent spacing, radius, colors, and typography.
- Use green for income or positive values.
- Use red or coral for expenses.
- If Reanimated already exists in the project, add subtle entrance animations for dashboard cards.
- Do not add Reanimated if it is missing.
- If Reanimated is unavailable, build the dashboard without animations and report it as a blocker or note.
- Update `progress/progress.md` with a short note that JOB-006 was completed.

## Constraints
- Do not add new packages.
- Do not change routing architecture.
- Do not modify auth behavior.
- Do not implement real data persistence.
- Do not create complex charts in this task.
- Do not overcomplicate the dashboard layout.
- Do not animate every element.
- Keep animations subtle and performance-friendly.
- Avoid unnecessary comments.
- Keep dashboard logic simple and readable.
- Preserve Expo Go compatibility.

## Definition of Done
- Dashboard placeholder is replaced with a real dashboard screen.
- Balance, income, and expense values are shown correctly from mock data.
- Category spending overview is visible.
- Recent transactions preview is visible.
- Income and expense states are visually distinguishable.
- Existing base UI system is used consistently.
- Reanimated is used only if already available.
- No unrelated files are changed.
- `progress/progress.md` is updated.
- Validation commands are run and results are reported.
