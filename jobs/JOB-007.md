# JOB-007 - Transactions List Screen

## Goal
Build the transactions tab screen that displays all mock income and expense transactions in a clean, scannable list.

## Why
The transaction list is one of the core screens of the finance tracker. Users need a clear way to review their financial activity, distinguish income from expenses, and open transaction details through the dynamic Expo Router route.

This task also connects the mock data foundation to route navigation through `/transaction/[id]`.

## Scope
This task should only focus on:
- Building the transactions list screen UI.
- Displaying all mock transactions.
- Showing income and expense visual states.
- Showing category and date information.
- Navigating to the dynamic transaction detail route when a transaction is tapped.
- Adding a simple empty state.
- Adding subtle list item animations if Reanimated is already available.

This task should not include:
- Transaction detail screen implementation.
- Add transaction form implementation.
- Real CRUD behavior.
- Filtering or search unless very simple and already supported by existing data.
- Backend integration.
- Auth logic changes.
- Dashboard changes.
- New package installation.

## Requirements
- Replace the transactions placeholder screen with a real transactions list UI.
- Use mock transactions from the existing mock data files.
- Use existing category data to display category names, colors, and icons when available.
- Use existing utility helpers for:
  - currency formatting
  - date formatting
  - transaction lookup or category lookup if available
- Display all transactions in a scrollable list.
- Each transaction item should show:
  - transaction title
  - category name
  - date
  - amount
  - income or expense state
- Expense amounts should be visually distinct from income amounts.
- Tapping a transaction should navigate to:
  - `/transaction/[id]`
- Use Expo Router navigation conventions.
- Add a clear empty state if there are no transactions.
- Use existing base components where appropriate:
  - `Screen`
  - `AppText`
  - `Card`
- Create a reusable `TransactionCard` or `TransactionListItem` component if it improves readability.
- Keep transaction rows easy to scan.
- Keep the screen modern, clean, premium, and consistent with dashboard styling.
- If Reanimated already exists in the project, add subtle entrance or list item animations.
- Do not add Reanimated if it is missing.
- If Reanimated is unavailable, build the list without animations and report it as a note.
- Update `progress/progress.md` with a short note that JOB-007 was completed.

## Constraints
- Do not add new packages.
- Do not change routing architecture.
- Do not modify dashboard behavior.
- Do not implement transaction detail UI in this task.
- Do not implement add/edit/delete transaction behavior.
- Do not implement backend or persistence.
- Do not introduce complex filters, charts, or analytics.
- Do not animate every small element unnecessarily.
- Keep animations subtle and performance-friendly.
- Keep logic simple and readable.
- Avoid unnecessary comments.
- Preserve Expo Go compatibility.

## Definition of Done
- Transactions placeholder is replaced with a real list screen.
- All mock transactions are displayed.
- Income and expense transactions are visually distinguishable.
- Category information is shown where available.
- Date and currency values are formatted.
- Tapping an item navigates to the dynamic transaction detail route.
- Empty state exists for no transactions.
- Existing base UI system is used consistently.
- Reanimated is used only if already available.
- No unrelated files are changed.
- `progress/progress.md` is updated.
- Validation commands are run and results are reported.
