# JOB-008 - Transaction Detail Screen

## Goal
Build the dynamic transaction detail screen using the Expo Router route `app/transaction/[id].tsx`.

## Why
The transaction detail screen is a core part of the app’s navigation architecture. It proves that dynamic routes work correctly and allows users to inspect a single income or expense transaction in more detail.

This task completes the flow from the transactions list to a detailed financial record.

## Scope
This task should only focus on:
- Building the transaction detail screen UI.
- Reading the transaction `id` from the dynamic route.
- Finding the matching mock transaction by ID.
- Showing full transaction details.
- Showing a clear not found state if the transaction does not exist.
- Adding subtle detail screen animations if Reanimated is already available.

This task should not include:
- Editing transactions.
- Deleting transactions.
- Add transaction form implementation.
- Real CRUD behavior.
- Backend integration.
- Dashboard changes.
- Transactions list changes unless a tiny navigation compatibility fix is required.
- Auth logic changes.
- New package installation.

## Requirements
- Replace the transaction detail placeholder with a real detail screen.
- Read the dynamic route parameter from Expo Router.
- Use the transaction `id` route param to find the matching transaction from mock data.
- Use existing utility helpers where appropriate:
  - transaction lookup by ID
  - currency formatting
  - date formatting
  - category lookup if available
- Display the transaction title clearly.
- Display the transaction amount prominently.
- Display whether the transaction is income or expense.
- Display category information:
  - category name
  - category icon if available
  - category color if available
- Display transaction date.
- Display transaction description if available.
- Display transaction ID only if useful for development or debugging; do not make it visually dominant.
- Use income and expense visual states consistently:
  - green for income or positive values
  - red/coral for expenses or negative values
- Add a back navigation option if the default stack header does not already provide one.
- Show a friendly “Transaction not found” state if the ID is missing or invalid.
- The not found state should offer a way to go back.
- Use existing base components where appropriate:
  - `Screen`
  - `AppText`
  - `Card`
  - `PrimaryButton`
- Create small reusable detail components only if they improve readability.
- Keep the UI modern, clean, premium, and easy to read.
- If Reanimated already exists in the project, add subtle entrance animations for the detail content.
- Do not add Reanimated if it is missing.
- If Reanimated is unavailable, build the detail screen without animations and report it as a note.
- Update `progress/progress.md` with a short note that JOB-008 was completed.

## Constraints
- Do not add new packages.
- Do not change routing architecture.
- Do not modify auth behavior.
- Do not modify dashboard behavior.
- Do not implement edit or delete actions.
- Do not implement real persistence.
- Do not introduce backend calls.
- Do not overcomplicate the layout.
- Do not animate every element unnecessarily.
- Keep animations subtle and performance-friendly.
- Keep logic simple and readable.
- Avoid unnecessary comments.
- Preserve Expo Go compatibility.

## Definition of Done
- Transaction detail placeholder is replaced with a real detail screen.
- Dynamic route parameter is read correctly.
- Matching mock transaction is displayed correctly.
- Missing or invalid transaction IDs show a friendly not found state.
- Transaction title, amount, type, category, date, and description are visible.
- Income and expense visual states are consistent with previous screens.
- Existing base UI system is used consistently.
- Reanimated is used only if already available.
- No unrelated files are changed.
- `progress/progress.md` is updated.
- Validation commands are run and results are reported.
