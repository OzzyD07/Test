# JOB-009 - Add Transaction Screen

## Goal
Build the add transaction screen with a usable MVP form for creating a new mock income or expense transaction.

## Why
Adding a transaction is one of the core user actions in a personal finance tracker. This task turns the add transaction tab into a functional form experience and prepares the app for future local persistence or real CRUD behavior.

For MVP, the form should focus on clean UI, local validation, and simple mock transaction creation behavior without backend integration.

## Scope
This task should only focus on:
- Building the add transaction screen UI.
- Creating a local form state for a new transaction.
- Adding simple validation for required fields.
- Allowing the user to choose income or expense.
- Allowing the user to choose a category from existing mock categories.
- Creating a mock transaction object on submit.
- Showing a success state or feedback after submit.
- Adding subtle form animations if Reanimated is already available.

This task should not include:
- Real backend integration.
- Persistent transaction storage.
- Editing existing transactions.
- Deleting transactions.
- Updating dashboard or transaction list data globally.
- Advanced form validation.
- Date picker package installation.
- New package installation.
- Major navigation changes.

## Requirements
- Replace the add transaction placeholder with a real form screen.
- Use existing base components where appropriate:
  - `Screen`
  - `AppText`
  - `Card`
  - `Input`
  - `PrimaryButton`
- The form should include:
  - transaction title input
  - amount input
  - income or expense selector
  - category selector
  - date input or simple date value
  - optional description input
- Amount input should use a numeric keyboard where possible.
- Transaction type selector should clearly distinguish:
  - income
  - expense
- Category selector should use existing mock categories.
- Category options should match the selected transaction type when practical.
- Basic validation should require:
  - title
  - valid amount greater than 0
  - transaction type
  - category
  - date
- Show readable validation messages.
- On submit, create a typed mock transaction object.
- The created transaction does not need to persist globally in this task.
- After submit, show a success message or reset the form.
- Keep the form modern, clean, premium, and easy to complete.
- Use consistent spacing, radius, colors, and typography.
- Use green for income selection.
- Use red or coral for expense selection.
- If Reanimated already exists in the project, add subtle form entrance or submit feedback animations.
- Do not add Reanimated if it is missing.
- If Reanimated is unavailable, build the form without animations and report it as a note.
- Update `progress/progress.md` with a short note that JOB-009 was completed.

## Constraints
- Do not add new packages.
- Do not implement a native date picker dependency.
- Do not implement real persistence in this task.
- Do not update mock data files dynamically.
- Do not modify dashboard or transaction list behavior.
- Do not implement edit/delete transaction behavior.
- Do not introduce backend calls.
- Do not change routing architecture.
- Keep validation simple and local.
- Keep form logic readable.
- Avoid unnecessary comments.
- Preserve Expo Go compatibility.

## Definition of Done
- Add transaction placeholder is replaced with a usable form screen.
- User can enter title, amount, type, category, date, and description.
- User can select income or expense.
- User can select a category from existing mock categories.
- Required validation messages are shown.
- Submit creates a typed mock transaction object.
- Submit gives visible success feedback or resets the form.
- Existing base UI system is used consistently.
- Reanimated is used only if already available.
- No unrelated files are changed.
- `progress/progress.md` is updated.
- Validation commands are run and results are reported.
