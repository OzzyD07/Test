# JOB-004 - Mock Auth with AsyncStorage

## Goal
Create a mock authentication system using React Context, a `useAuth` hook, and AsyncStorage session persistence.

## Why
The app needs a simulated authentication flow before building real login and register screens. This task allows the app to remember a mock user session locally and decide whether the user should see auth screens or the main tab layout.

This keeps the MVP Expo Go friendly while still testing a realistic app structure.

## Scope
This task should only focus on:
- Creating an auth context/provider.
- Creating a `useAuth` hook.
- Creating mock login, register, logout, and session restore logic.
- Using AsyncStorage to persist mock session state.
- Connecting the root routing decision to auth state if needed.

This task should not include:
- Final login/register UI design.
- Dashboard implementation.
- Transaction list implementation.
- Transaction detail UI.
- Add transaction form.
- Real backend authentication.
- Password security implementation.
- Token refresh logic.
- Reanimated animations.

## Requirements
- Create an auth provider using React Context.
- Create a `useAuth` hook that exposes auth state and actions.
- Auth state should include:
  - current user
  - loading state
  - error state if useful
  - authenticated status
- Implement mock `login` action.
- Implement mock `register` action.
- Implement `logout` action.
- Persist mock session data using AsyncStorage.
- Restore mock session from AsyncStorage when the app starts.
- Clear persisted session on logout.
- Use the existing `User` type if available.
- If additional auth-specific types are needed, add them in `types/`.
- Keep credentials mock-only and simple.
- Do not store or validate real passwords.
- Update app-level layout or entry route only as much as needed to provide auth context and route users correctly.
- If no session exists, the user should be directed to the auth flow.
- If a session exists, the user should be directed to the main tab flow.
- Keep route changes compatible with the structure created in JOB-001.
- Update `progress/progress.md` with a short note that JOB-004 was completed.

## Constraints
- Do not add new packages.
- Use AsyncStorage only if it already exists in the project.
- Do not modify `package.json` to add AsyncStorage.
- Do not implement production-grade auth.
- Do not connect to a real backend.
- Do not implement final auth screen styling in this task.
- Do not create transaction-related logic.
- Do not introduce global state libraries such as Redux or Zustand.
- Keep the provider and hook simple and readable.
- Avoid unnecessary comments.
- Preserve Expo Go compatibility.

## Definition of Done
- Auth context/provider exists.
- `useAuth` hook exists and works through the provider.
- Mock login persists a session.
- Mock register persists a session.
- Logout clears the session.
- App startup restores the saved session from AsyncStorage.
- The app can route users based on mock auth state.
- Missing AsyncStorage dependency is reported as a blocker instead of modifying `package.json`.
- No unrelated files are changed.
- `progress/progress.md` is updated.
- Validation commands are run and results are reported.
